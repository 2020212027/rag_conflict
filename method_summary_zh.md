## 基于文档依赖性分析的 RAG 冲突信号还原方法

### 1. 问题定义

检索增强生成（RAG）系统在面对信息冲突时，依赖一个关键的隐含假设：**检索到的每篇文档代表一个独立的信息源**。当多篇文档支持同一答案时，LLM 将其视为来自多个独立来源的交叉验证，从而赋予该答案更高的置信度。

然而，这一假设在现实中系统性地不成立：
- Web 语料库中大量文档是同源改写、转载或自动化生成的副本
- 检索器（如 BM25、DPR）不考虑文档间的依赖关系
- 结果：**非独立的文档被当作独立证据，冲突信号被系统性地扭曲**

**核心问题**：

```
真实场景：检索返回 10 篇文档
  8 篇说"答案是 Paris"（实为同一来源的 8 个改写版本）
  2 篇说"答案是 Lyon"（独立来源）

LLM 感知到的冲突信号：
  未处理时：  8 vs 2 → LLM 认为 Paris 有压倒性多数证据 → 选 Paris（错误）
  处理之后：  1 vs 2 → LLM 发现 Lyon 有更多独立支持 → 选 Lyon（正确）
```

**问题本质**：文档冗余制造了"虚假共识"（False Consensus）——LLM 将"同一来源说了 8 遍"误判为"8 个独立来源都这么说"，导致本应是 1:2 的真实冲突被扭曲为 8:2 的虚假一致。

**我们的目标**：在 LLM 处理冲突之前，通过文档依赖性分析识别并消除冗余证据，**还原冲突的真实独立信息源比例**。

---

### 2. 评估设计

为量化"虚假共识"对 RAG 准确率的影响，以及依赖性分析对冲突信号还原的效果，构建两组平行评估集（各 N=215）：

| 数据集 | 组成 | 模拟场景 | 评估目的 |
|--------|------|----------|----------|
| **amp_8** | 每个查询 6 篇干净文档 + 8 篇同源改写 | 极端冗余下冲突信号被扭曲 | 量化信号还原后 LLM 判断准确率的恢复 |
| **clean** | 每个查询 14 篇独立检索文档 | 无冗余的正常检索 | 验证方法不会误将独立来源标记为冗余 |

**主要指标**：归一化精确匹配率（NEM）——生成答案是否包含金标准答案。

**评估维度**：
- **Δ NEM (amp_8)**：处理后 NEM 提升 → 衡量冲突信号还原后 LLM 判断能力的恢复
- **Δ NEM (clean)**：处理后 NEM 变化 → 衡量独立信息源被误合并的代价（理想值 ≈ 0）

---

### 3. 方法：基于文档依赖性分析的冲突前处理

本方法作为检索与生成之间的预处理步骤，目标是在 LLM 处理冲突之前，**识别哪些文档代表同一信息源，将冲突比例还原为真实的独立信息源之间的比例**。包含两个核心层次：

#### 3.1 第一层：词汇级依赖候选识别（零成本）

**目标**：快速识别可能存在依赖关系的文档对，避免对所有 O(n²) 文档对进行昂贵的 LLM 调用。

**方法**：对每篇文档前 800 字符进行分词（小写化、去停用词），计算 token 级 Jaccard 相似度：

```
Jaccard(A, B) = |tokens(A) ∩ tokens(B)| / |tokens(A) ∪ tokens(B)|
```

**阈值**：Jaccard ≥ 0.20

**阈值选择依据**（在 80 对放大 + 80 对独立文档对上验证）：
- 阈值 0.20 时：捕获 72.5% 的放大文档对，仅放过 6.2% 的独立文档对
- 第一层单独精确率：92.1%
- 尽管 72.5% 的配对级召回率看似一般，但**聚类级召回率接近 100%**：8 个节点的连通图仅需 7 条边，而 72.5% × C(8,2)=28 ≈ 20 条边，远超最小生成树需求

**计算开销**：O(n²) 字符串操作，零 API 调用。对于 n=14 篇文档，评估 91 对耗时 <10ms。

#### 3.2 第二层：配对依赖性分类（LLM 驱动）

**目标**：对通过第一层的候选文档对，精确判断二者是否代表同一信息源——即它们是否构成"独立证据"。

**方法**：使用校准后的 LLM 提示词（GPT-4o-mini，temperature=0）将每对文档分为四级依赖量表：

| 级别 | 定义 | 信息源解读 | 处理方式 |
|------|------|-----------|----------|
| **D0** | 完全无关 | 两个无关信息源 | 保留两篇 |
| **D1** | 同一话题，独立写作 | 两个独立信息源（有效证据×2） | 保留两篇 |
| **D2** | 部分依赖（共享来源但有显著独立内容） | 部分重叠但携带增量信息 | 保留两篇 |
| **D3** | 高度依赖（逐句改写/同源复制） | 同一信息源（仅算 1 条证据） | **合并为一** |

**关键设计决策**：仅 D3 触发合并。D2 文档对被保留，因为它们虽共享来源但可能包含事实不同的增量信息——合并它们会消除真实的冲突信号（实验验证：批量方法中 26% 的误伤案例源于错误合并 D2 级文档对）。

**提示词设计原则**：
1. 显式校准示例（关于同一主题的两篇维基百科文章 = D1，而非 D3）
2. 要求"罕见共享细节"作为 D3 证据——仅主题相同不足以判定
3. 结构化 JSON 输出确保可靠解析

**验证性能**（80 对放大 + 78 对独立文档对）：
- D3 召回率 P(模型=D3 | 真实放大)：53.8%
- D3 精确率 P(真实放大 | 模型=D3)：95.6%
- 组合误报率（第一层 × 第二层）：6.2% × 2.6% ≈ **0.16%/对**

#### 3.3 信息源识别与代表选择：Union-Find 聚类

**目标**：将配对级 D3 判断聚合为"同一信息源"的文档簇，每个簇 = 1 个独立信息源，为每个信息源选择最佳代表文档。

**方法**：
1. 对 top-N 检索文档初始化 Union-Find 数据结构
2. 对每个被第二层判为 D3 的文档对，合并到同一连通分量
3. 每个连通分量代表一个独立信息源：
   - 大小 = 1：该文档即为唯一的信息源代表
   - 大小 > 1：选择**最长文档**作为该信息源的代表，移除其余副本
4. 输出去重后的文档集，其中每篇文档代表一个独立的信息源

**代表选择策略（最长文档）的理由**：
- 较长文档通常包含更完整的信息，是该信息源的最佳代表
- 在同源改写场景中，所有副本语义等价，任何选择标准均不影响冲突信号比例
- 在自然近似重复中，较长版本通常包含较短版本的全部信息

**聚类级召回率分析**：
- 第一层放过 ~20/28 条同源边；第二层确认 ~11 条为 D3
- 8 个同源节点仅需 7 条边实现完全连通 → 信息源识别率 ≈ 100%
- 观测到的聚类大小：均值=5.4，最大=10（amp_8 数据，真实同源数为 8）

### 4. 端到端流程

```
输入：查询 Q，检索文档集 D = {d₁, d₂, ..., d_N}
                    │
                    ▼
┌───────────────────────────────────────────────────┐
│  第一层：依赖候选识别（Jaccard 粗筛）              │
│  对每对文档 (dᵢ, dⱼ)：                            │
│    若 Jaccard(dᵢ, dⱼ) ≥ 0.20 → 加入候选对集合    │
│  开销：O(n²) 字符串操作，零 API 调用              │
└───────────────────────────────────────────────────┘
                    │
          候选对（可能存在依赖关系的文档对子集）
                    │
                    ▼
┌───────────────────────────────────────────────────┐
│  第二层：文档依赖性精确判定（D0–D3 分类）          │
│  对候选对中的每对 (dᵢ, dⱼ)：                      │
│    level = LLM_依赖性分类(Q, dᵢ, dⱼ)             │
│    若 level == "D3"：union(dᵢ, dⱼ)               │
│          （标记为同一信息源）                      │
│  开销：每个候选对 1 次 API 调用                    │
└───────────────────────────────────────────────────┘
                    │
                    ▼
┌───────────────────────────────────────────────────┐
│  信息源聚合与代表选择                              │
│  每个连通分量 = 1 个独立信息源                     │
│  对每个簇 C（|C| > 1）：                          │
│    代表 = argmax_{d ∈ C} len(d)                   │
│    移除 C 中非代表文档                            │
│  输出：D' ⊆ D（每篇文档 = 1 个独立信息源）        │
└───────────────────────────────────────────────────┘
                    │
                    ▼
┌───────────────────────────────────────────────────┐
│  RAG 生成（基于独立信息源的冲突处理）              │
│  答案 = LLM(Q, D')                               │
│  此时 LLM 看到的文档间冲突比例 = 真实的独立源比例  │
└───────────────────────────────────────────────────┘
```

### 5. 实验结果

#### 5.1 主要结果

| 条件 | 原始 NEM | 处理后 NEM | Δ NEM | 平均合并数 | 依赖性分析次数 |
|------|----------|-----------|-------|-----------|---------------|
| **amp_8**（冲突信号被扭曲） | 30.7% | 43.7% | **+13.0pp** | 4.8 | 4,777 |
| **clean**（冲突信号正常） | 70.7% | 66.5% | −4.2pp | 0.8 | 421 |

**解读**：在冲突信号被严重扭曲的场景（amp_8）下，信息源还原使 LLM 判断准确率恢复了 13.0 个百分点。在冲突信号本身正常的场景（clean）下，方法的副作用为 4.2pp——因少量独立信息源被误判为同源。

#### 5.2 与批量分类基线对比

| 指标 | 批量二分类 | 分层依赖性分析 | 改善 |
|------|-----------|---------------|------|
| amp_8 Δ NEM | +11.6pp | **+13.0pp** | 信号还原能力提升 12% |
| clean Δ NEM | −6.0pp | **−4.2pp** | 独立源误合并减少 30% |
| amp_8 平均合并数 | 6.2 | 4.8 | 信息源识别更精准 23% |
| clean 平均合并数 | 1.9 | 0.8 | 独立源误合并减少 58% |

**分析**：批量方法将依赖性判断压缩为二分类（重复/不重复），无法区分"同名实体的不同事件报道"（D1）与"同一来源的改写"（D3），导致大量独立信息源被错误合并。分层方法通过四级量表精确刻画依赖关系，仅合并真正的同源文档。

#### 5.3 翻转分析

| | 正向翻转（错→对） | 负向翻转（对→错） | 比值 |
|--|-------------------|-------------------|------|
| **amp_8** | 35 | 7 | 5.0:1 |
| **clean** | 6 | 15 | 0.4:1 |

**解读**：
- **amp_8**：冲突信号还原后，35 个此前被虚假共识误导的查询恢复了正确判断，仅 7 个查询因信息源识别错误而受损。5:1 的正负比表明方法整体方向正确。
- **clean**：15 个负向翻转中，少量独立信息源被误判为同源，导致 LLM 丧失了原本的多源交叉验证能力。这是方法的主要改进方向。

#### 5.4 信息源聚类统计

| 数据集 | 聚类数量 | 平均大小 | 最大大小 |
|--------|---------|---------|---------|
| **amp_8** | 232 | 5.4 | 10 |
| **clean** | 149 | 2.1 | 6 |

**解读**：
- **amp_8** 中聚类均值 5.4（真实同源数为 8），说明方法成功将大部分同源文档归入同一信息源
- **clean** 中聚类均值仅 2.1，说明方法在正常检索场景下较少产生大规模误合并
- 聚类大小的显著差异（5.4 vs 2.1）本身即为"文档集是否存在冗余导致的虚假共识"的有效诊断信号

### 6. 开销分析

| 组件 | amp_8（高冗余，每查询） | clean（无冗余，每查询） |
|------|----------------------|----------------------|
| 第一层（Jaccard 候选识别） | ~91 对，0 次 API 调用 | ~91 对，0 次 API 调用 |
| 第二层（依赖性分类） | ~22 次 API 调用 | ~2 次 API 调用 |
| RAG 生成 | 2 次调用（处理前后对比） | 1–2 次调用 |
| **总 API 开销** | ~24 次/查询 | ~3 次/查询 |
| **端到端耗时** | ~47 秒/查询 | ~6 秒/查询 |

**设计亮点**：分层架构具有天然的"按需计算"特性。当文档集无冗余时（clean），第一层即过滤掉绝大部分文档对，第二层仅需处理约 2 对——开销接近于零。仅当检测到大量词汇重叠文档对时，才触发更多依赖性分析。这意味着正常查询几乎不受延迟影响。

### 7. 关键设计决策与消融分析

| 设计决策 | 理由 | 实验证据 |
|----------|------|----------|
| Jaccard ≥ 0.20（而非余弦相似度） | Jaccard 在低阈值区间对依赖关系的判别力更强；零成本 | Jaccard 精确率 92.1% vs 余弦 74.4%（相近召回率下） |
| 仅合并 D3（不含 D2） | D2 文档对虽共享来源但可能携带冲突信息，合并会消灭真实冲突信号 | 批量方法 19 个负向翻转中，26% 源于错误合并 D2 级别文档对 |
| 保留最长文档作为信息源代表 | 最长版本通常信息最完整，是信息源的最佳代表 | 同源改写场景下各副本语义等价；自然重复中长版本包含短版本全部信息 |
| 配对四级分类（而非批量二分类） | 批量二分类将"同名实体不同事件"与"同源改写"混为一谈，导致独立信息源被错误合并 | 批量方法 58% 误伤源于将关于同一实体的独立报道误判为同源文档 |

---

### 8. 核心贡献总结

1. **揭示问题**：发现 RAG 系统中的文档冗余会系统性地扭曲冲突信号，导致 LLM 对同源信息产生虚假的高置信度（amp_8 场景下 NEM 从 70.7% 暴跌至 30.7%，下降 40pp）。

2. **提出方法**：设计了基于文档依赖性分析的冲突前处理流水线，在 LLM 处理冲突之前识别并消除冗余证据，使冲突检测回归真实的独立信息源比例。

3. **验证效果**：在极端冗余场景下恢复了 +13.0pp 的判断准确率，同时将正常场景的副作用控制在 −4.2pp 以内。

---

### 9. 局限性与未来工作

1. **残余独立源误合并（−4.2pp）**：虽较批量基线（−6.0pp）显著改善，在正常检索场景下仍有少量独立信息源被误判为同源。未来可探索置信度校准阈值或簇大小门控策略（仅簇大小 ≥ 3 时触发合并，因自然冗余通常为 2 篇）。

2. **延迟**：第二层在高冗余场景下增加约 22 次 LLM 调用。可通过并行 API 调用或微调轻量级依赖性分类器（如 DeBERTa）替代提示词 LLM 来缓解。

3. **对抗性改写**：感知到词汇级粗筛的攻击者可能生成 Jaccard 重叠度更低的改写。未来应评估对抗性改写策略下的鲁棒性，或引入语义级粗筛（如句向量余弦）作为补充。

4. **软权重替代方案**：当前方法采用硬合并策略（同源文档仅保留一篇）。更精细的方案是利用依赖性信息在 RAG 提示词中对冗余文档降权（如逆簇大小加权），在抑制虚假共识的同时保留可能有用的冗余信息。
